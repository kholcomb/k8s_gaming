What to Look For:
==================

Users are randomly getting logged out of your application. They successfully log in, 
but their next request shows them as not authenticated. This is a classic sign of 
session management issues in a distributed system.

Check These Areas:
------------------

1. **Client Logs**
   Check what responses the client is getting:
   ```bash
   kubectl logs client -n k8squest
   ```

   Look for:
   ‚Ä¢ Are responses coming from different pods?
   ‚Ä¢ "Session Pod 1", "Session Pod 2", "Session Pod 3" alternating?

2. **Service Configuration**
   View the Service:
   ```bash
   kubectl describe service session-service -n k8squest
   ```

   Look for:
   ‚Ä¢ Session Affinity: None (or missing)
   ‚Ä¢ This means load-balancing is RANDOM

3. **The Problem**
   ```
   Request 1: User logs in ‚Üí Goes to Pod 1 ‚Üí Session stored in Pod 1
   Request 2: User accesses page ‚Üí Goes to Pod 2 ‚Üí No session! Logged out!
   Request 3: User tries again ‚Üí Goes to Pod 3 ‚Üí No session! Still logged out!
   Request 4: User tries again ‚Üí Goes to Pod 1 ‚Üí Session found! Logged in!
   ```

üéØ Key Questions:
-----------------

‚Ä¢ How does the Service decide which pod to send each request to?
‚Ä¢ What happens to session data when a request goes to a different pod?
‚Ä¢ Is there a way to ensure the same client always goes to the same pod?

üí≠ Think About:
---------------

Imagine you're at a restaurant with 3 cashiers. You order from Cashier 1, but when 
you come back to pick up your food, you're randomly assigned to Cashier 2. Cashier 2 
has no record of your order!

Solution? Make sure you always go back to the SAME cashier (sticky sessions).

In Kubernetes, this is called "session affinity" or "sticky sessions."
